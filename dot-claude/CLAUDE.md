# 用户个人偏好配置

## 语言
- 所有交互使用**简体中文**
- 技术术语可以保留英文

## 模型偏好
- 默认使用 DeepSeek API 提供支持
- 日常任务：`deepseek-chat`
- 复杂推理：`deepseek-reasoner`

## 全局规则
- 不要询问过多确认，确认框架后直接执行
- 优先使用文件操作工具而非 shell 命令
- 重要决策必须记录到项目记忆系统
- 不允许修改 API 密钥和关键安全配置

## 常用设置
- 主题：暗色 (dark)
- 关闭不必要的流量以节省 API 成本

## 工作流指令（覆盖默认行为）

### 1. 任务路由规则

收到任务时按以下优先级路由：

| 任务类型 | 工具/技能 | 说明 |
|---------|----------|------|
| 需求分析/规划 | `/plan` 命令 → TaskCreate | 先拆解再执行 |
| 代码生成 | 直接编写 + `review-deep` 审查 | 生成完毕自动审查 |
| Bug 修复 | `/debug` 命令 | 结构化流程，输出复盘 |
| 代码审查 | `/review-deep` 命令 | 并行多维度审查 |
| 发布部署 | `/create-release` 或 `/deploy-release` | 标准发布流程 |
| 简单的查询/解释 | 直接回答 | 无需路由 |

### 2. 多智能体协作协议

- 当任务涉及多个独立维度（如：前端+后端，逻辑+安全），**必须**使用 Agent 工具并行处理
- 复杂重构/多文件修改：主进程拆解步骤 → 后台 Agent 并行执行 → 主进程整合审查
- 代码审查场景：同时启动逻辑审查 + 安全审查 + 性能审查三个独立 Agent
- 子任务完成后，主进程负责合并结果和最终审查

### 3. 记忆写入协议

以下情况**必须**写入 memory/：

- **架构决策**：选择了某种方案而放弃其他方案的原因
- **踩坑记录**：遇到的 Bug、根因、修复方案
- **项目约定**：非显而易见的命名规则、代码风格约定
- **工具配置变更**：settings.json、Hooks 的重大变更

写入格式：
```markdown
---
name: short-descriptive-slug
description: 一句话总结
metadata:
  type: reference|project|feedback
---

内容主体（包含 What + Why）
```

### 4. 执行原则
- 确认大框架后直接执行，无需逐条确认
- 优先使用 Edit/Write/Read 工具而非 shell 命令
- 完成每个子任务后更新 TaskCreate 状态
- 重要变更结束后检查是否满足记忆写入条件
